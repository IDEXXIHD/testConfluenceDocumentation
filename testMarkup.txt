******** ******** author: mcodere

  !!!!!!!!
[mh]  Note:  I'm doing something different here.  
      PimsDomainObjectResolver does not behave well when wrapped in a transaction.
      The dao's it calls add stuff to the persistent context and this causes it to do an unfortunate flush and error.
      Luckily it is not wrapped in a transaction when running normally.
      So, to maintain that, this test is also not wrapped in a transaction.
      Which is fine, but we just need to remember to clean up after ourselves (delete created patients/clients)
      when we're done with the test.
      Alternately (possibly better), we could just fix PimsDomainObjectResolver so that it worked with/without a transaction.
      Note:  a side benefit of not running in a transaction is we can do real queries of the db.  Inside of a transaction we will get
      the entities that are already stored in our persistent context back when querying instead of new entities from the db.

  !!!!!!!!
